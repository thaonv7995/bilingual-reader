242 CHƯƠNG 8 Kiến trúc hướng sự kiện với Spring Cloud Stream

Việc publish message thực sự diễn ra trong method publishOrgChange(). Method này tạo một Java POJO có tên OrganizationChangeModel. Tôi sẽ không đưa code của OrganizationChangeModel vào chương vì class này không gì hơn một POJO chứa ba phần tử dữ liệu:

  • Action—Đây là hành động đã kích hoạt sự kiện. Tôi đã đưa action vào message để cung cấp thêm ngữ cảnh cho message consumer về cách xử lý sự kiện.
  • Organization ID—Đây là organization ID gắn với sự kiện.
  • Correlation ID—Đây là correlation ID của lời gọi dịch vụ đã kích hoạt sự kiện. Bạn nên luôn đưa correlation ID vào các sự kiện vì nó giúp rất nhiều trong việc theo dõi và debug luồng message qua các dịch vụ.

Khi bạn đã sẵn sàng publish message, hãy dùng method send() trên class MessageChannel được trả về từ method source.output().

Method send() nhận một class Spring Message. Bạn dùng một helper class của Spring có tên MessageBuilder để lấy nội dung của class OrganizationChangeModel và chuyển đổi thành class Spring Message.

Đây là toàn bộ code bạn cần để gửi một message. Tuy nhiên, ở thời điểm này, mọi thứ có thể khiến bạn cảm thấy hơi như phép thuật vì bạn chưa thấy cách bind organization service tới một message queue cụ thể, chưa nói tới message broker thực sự. Tất cả được thực hiện thông qua cấu hình. Listing 8.3 trình bày cấu hình ánh xạ Spring Cloud Stream Source của dịch vụ tới message broker Kafka và một topic message trong Kafka. Thông tin cấu hình này có thể đặt trong file application.yml của dịch vụ hoặc trong mục Spring Cloud Config cho dịch vụ.
