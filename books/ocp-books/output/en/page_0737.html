The logic to determine the properties files needed to create a resource bundle is important and I will explain it shortly, but in this example, since we don't have any locale specific properties file, the getBundle method just loads a properties file with the same name as that of the resource bundle (but with extension .properties), i.e., helloapp.properties.

3. The name of the resource bundle and the corresponding properties file follows the same convention as the one followed by classes. Therefore, while locating a bundle named helloapp, the getBundle method expects helloapp.properties to be located directly on the classpath. If the name of the bundle were test.helloapp, then helloapp.properties would have to be present in the test folder and the test folder would have to be on the classpath.

4. The name of the keys in the properties file can be anything but the code must use the same key to loop up the value. Therefore, it is a common practice to use strings containing text that makes sense in the language that the dev team understands as keys. Using keys such as key1, key2, makes it hard to tie a key to the text that is meant to be shown to the user for that key.

5. The backslashes (\) in the properties files are required to escape the space character.

24.3.2 Understanding Resource Bundles☝

Resource bundles are hierarchical, meaning, a resource bundle always has a parent bundle unless the bundle is at the top of the hierarchy, in which case, it is called the "base bundle". The hierarchy of resource bundles is established using a set of resource bundles whose names start with a common base name, such as helloapp used in the previous example.
