Managing exceptions encountered while closing resources☝

The exact mechanism of resource closure is a bit more complicated than just wrapping the code in a try/finally block. Specifically, this simplification does not explain what happens if the close method of one resource throws an exception. Are the rest of the resources still closed? What if more than one resource throws an exception while closing? What if the code in the try block also throws an exception - which exception does the caller receive? Although the OCP 17 exam does not expect you to know these details, the OCP 21 exam does.

Along with the introduction of the try-with-resources statement, Java 7 also added the concept of “suppressed exceptions”, which is utilized to address the issues that I just mentioned.

Suppressed exceptions is an array of Throwables that can be used to store multiple Throwable objects within a Throwable object. The benefit of maintaining this array is that you can perform multiple tasks in a method and also remember all the exceptions that are thrown while performing those tasks before throwing that single exception from the method. The caller of the method can look at the array of suppressed exceptions and know of all the exceptions that were generated during the execution of the method.
