24.1 Understanding localization☝

While developing an application it is easy to forget that the application may be used by people in different countries with different languages. At a bare minimum, an application is expected to display information meant for the consumption of the user in the language that the user can understand. An application usually produces user consumable information in the form of UI control labels such as OK and Cancel buttons, pop up messages, and, any pro-forma text that goes in the output produced upon the user's request such as PDF/Word reports. Since the regions and languages of the users of the application cannot always be known at the time of development of the application, it is almost impossible to hardcode region specific formats and texts in the application itself. Thus, instead of trying to do that, it is better for the application to not assume any language at all and use the facilities of the run time environment to localize the messages. The runtime environment can easily format information such as dates and numbers as per the local format but it cannot translate arbitrary application specific messages to the local language. To manage that, the application must use message "keys" that can be looked up into a region and language specific "resource bundle" at runtime and use the returned values to generate the messages.
