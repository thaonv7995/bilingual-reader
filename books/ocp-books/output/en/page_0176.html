15.1.3 Pieces of the exception handling puzzle☝

Java exception handling mechanism comprises several moving parts. What makes this mechanism a bit complicated is that you need to put each part in its right place to make the whole thing work. So, let me introduce these parts first briefly.

The java.lang.Throwable object -☝

Throwable is the root class of all exceptions. It captures the details of the program and its surroundings at the time it is created. Among other things, a Throwable object includes the chain of the method calls that led to the exception (known as the “stack trace”) and any informational message passed by the programmer as a String argument while creating that exception (which can be retrieved using its getMessage() method). This information is helpful in determining the location and the cause of the exception while debugging a program.

You may have seen crazy looking output on the console containing method names upon a program crash. This output is actually the stack trace contained in the Throwable object. For example, if m1() calls m2(), m2() calls m3() and the code in m3() throws an exception, then the stack trace of that exception will contain the names of the methods m3, m2, and m1, in that order.

Throwable has two subclasses - java.lang.Error and java.lang.Exception, and a huge number of grand child classes. Each class is meant for a specific situation. For example, a java.lang.NullPointerException is thrown when the code tries to access a null reference or an java.lang.ArrayIndexOutOfBoundsException is thrown when you try to access an array beyond its size.
