  1. In the case of primitive return types, the Java Language Specification mandates that the return type of the overriding method must match exactly to the return type of the overridden method even though it does define supertype-subtype relationships between numeric types.
  2. Parameters - The list of parameters that the overriding method takes must match exactly to that of the overridden method in terms of types and order (parameter names don’t matter). Indeed, if there is a difference between the types and/or order of parameters, it would not be an override but an overload. I have already discussed overloading of methods.
  3. Throws clause - This rule will make more sense once you learn about exception handling but for now just remember that an overriding method cannot put a wider checked exception (i.e. a superclass exception) in its throws clause than the ones present in the throws clause of the overridden method. For example, if the throws clause of the overridden method has IOException, the overriding method cannot have Exception because Exception is a superclass of IOException but it may have FileNotFoundException because FileNotFoundException is a subclass of IOException and is thus, narrower than IOException.

The overriding method cannot throw a new checked exception that is not listed in the throws clause of the overridden method either. The overriding method may decide to not have a throws clause altogether though.

Note that these rules apply only to checked exceptions because these are the only ones the compiler cares about.
