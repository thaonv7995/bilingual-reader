Scenario 3: The first thread started by main runs and gets the lock upon calling rlck.lock(). The scheduler now suspends the first thread and runs the second thread. The second thread calls rlck.tryLock() but this call will return false because the rlck is already locked by the first thread. So, the second thread executes rlck.unlock() in the else part. 20.5.3 Using ReadWriteLock and ReentrantReadWriteLock☝

Blocking simultaneous access from multiple threads to a shared field makes sense only if that shared field is modified. As you learned in the previous section, an immutable field poses no risk when read by multiple threads simultaneously. So, if there is a shared field that is read more frequently than it is written to, it would be very inefficient to block access to that field for every read operation. It would be ideal to block access to it only when a thread actually wants to modify it.

Well, this is the functionality that a ReadWriteLock provides. It allows multiple reads of a field to execute simultaneously but blocks all access to the field from other threads when one thread wants to modify it.

In technical terms, a ReadWriteLock has two locks - a “read” lock, which can be acquired by any number of threads simultaneously while the write lock is free and an exclusive “write” lock, which can be acquired by only one thread at a time and only when no thread has a read lock.
