Serializable clause. To use Java's inbuilt serialization mechanism the class that we want to serialize must be "serializable", i.e., it must directly or indirectly implement the java.io.Serializable interface. Not only the class but its instances fields must also be serializable. This is a recursive requirement. So, if a class has an instance field of some other class, then that other class must also implement Serializable and that other class's instance fields' types must also do the same. It nests as deep as there are instance fields. If any object that is not serializable is found while serializing an object of a class then a java.io.NotSerializableException, which is an IOException, is thrown by the writeObject method.

Since static fields do not constitute the state of an object, they are never serialized. So, it doesn't matter whether a static field is serializable or not.

Fortunately, all primitives are serializable and most of the commonly used classes from the Java SDK such as String and wrapper classes (Integer, Double, etc.) implement Serializable. Arrays of primitives and serializable objects are serializable too. Thus, in the above code, the values of x, y, and title will be preserved in the serialized data. Note that java.lang.Object class does not implement Serializable. If it did, everything would be serializable!

Marker interface

The Serializable interface doesn't define any field or method. Its sole purpose is to indicate to the JVM that objects of this class are intended to be serialized and deserialized. Thus, it is just a "marker" interface.

Marking a class serializable is important from security perspective because an object's state may comprise confidential information and the developer may not want it shared outside of the JVM in which the application is running. Once an object is serialized and that serialized data goes out of the JVM, for example, once it is saved in a file, the JVM has no control over it. Anyone can tamper with it and in that case, recreating that object using tampered data may pose a serious security risk to the application. Implementing Serializable indicates to the JVM that we are okay with the risks involved in serializing and deserializing objects of this particular class.
