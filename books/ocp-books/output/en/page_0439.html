The transient keyword☝

If there are instance fields of a class that we don't want to be present in the serialized data, such as the securityKey field in the above code, we need to mark them transient. The serialization mechanism ignores all transient fields of a class while serialization. Since the serialized data doesn't contain any value for transient fields, such fields are initialized to their default values (i.e. 0, false, or null) while deserializing the data. Observe in the above output that the securityKey field is null after deserialization.

Role of constructors and instance initializers during deserialization☝

You might have observed from the above output that the print statement from the constructor is not executed when the AppPreferences object is deserialized. That is because constructors and instance initializers of a serializable class are not executed when an object of that class is created through deserialization. The reason is simple. The purpose of constructors and instance initializers is to initialize the values of instance fields to sensible values as per the business logic of the class. However, these values may change during program execution. Thus, at the time of serialization, the values of the instance fields may not be the same as the values that were given to those fields by the constructor and/or instance initializers. Therefore, it doesn't make sense to execute constructors and instance initializers when we are trying to set the values of the fields of the object from the serialized data.
