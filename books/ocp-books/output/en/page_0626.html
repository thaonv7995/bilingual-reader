Trying to create a ZoneId for a non-existing place such as ZoneId.of("Somewhere/Timbaktoo"); will cause a java.time.zone.ZoneRulesException to be thrown with a message, Unknown time- zone ID: Somewhere/Timbaktoo.

ZonedDateTime: A ZonedDateTime object represents a date and time in a particular time zone. Thus, it needs two things - the number of milliseconds elapsed since the epoch and the ZoneId of the time zone whose time it represents. It uses the ZoneId to fetch all the rules that affect the date/time used in that zone and applies them to the show you the regional time.

In layman terms, for example, if people in a particular zone shift their clocks backward or forward on certain dates in a year (such as for saving daylight), then a ZonedDateTime object created for that zone knows those dates and the amount of shift that needs to be applied to show the correct time in that region at any instant. Since a ZonedDateTime pulls all applicable rules for that zone using the ZoneId, a ZoneId is necessary to create a ZonedDateTime.

OffsetDateTime: - The OffsetDateTime class is a bit weird because it neither represents a local datetime nor does it represent a datetime in any particular geographical region. It actually contains just two things - the number of milliseconds elapsed after the epoch and an offset amount called “zone offset”, which is nothing but the amount by the which time represented by this OffsetDateTime differs from UTC. Since an OffsetDateTime does not represent time for any specific geographical region, it does not care about any region specific rules. In a way, an OffsetDateTime object represents a date/time of an imaginary place that is always “offset” from the UTC by the given offset value.
