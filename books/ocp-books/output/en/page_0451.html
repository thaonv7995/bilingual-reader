The same reasoning applies to the input stream of a program. A program’s input source may not always be a console and so, the input stream must be capable of delivering binary data to the program. If a program knows that it is always going to get data from the console as its input, it can either wrap the input stream into a Reader (as shown above) or use the Console class, which you will see next.

19.4.2 Using java.io.Console☝

The Console class provides a higher level abstraction for the console associated with the Java program. As you will observe from the examples below, it is a bit more convenient to use and also has a few advantages over using System.in and System.out directly.

Acquiring the Console object☝

The Console class does not have any public constructor and so you cannot create a Console object using new. There is only one Console object for the entire application and you must acquire it by calling System.console(). The important thing to know about this method is that it returns null if no console is associated with the JVM. This may happen, for example, when an application is launched in the background using javaw. Unfortunately, console is not available even when you you run your program in IDEs such as Netbeans and Eclipse because these IDEs too execute the program using a background process. But they do redirect the System.in and System.out to a window in the IDE, so you can still use the streams directly for printing and accepting user input.
