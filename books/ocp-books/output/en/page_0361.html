How a Spliterator partitions the elements depends on how the Spliterator is coded by the class of the underlying collection.

Although every detail of the implementation cannot be expressed programmatically, important features of a Spliterator can be inspected programmatically using a few static final int constants and the hasCharacteristics(int characteristics) method defined in Spliterator. For example, split1.hasCharacteristics(Spliterator.ORDERED); returns true, which means that this Spliterator has a definite order in which it keeps its elements. This is expected because the underlying collection in this example is a List, which is ordered. If you run the same example using a HashSet instead of a List, it will print false.

You can also check if a Spliterator knows the exact number of elements in it by passing Spliterator.SIZED to hasCharacteristics and if not, you may call estimateSize() to get an estimate. Knowing the size is helpful in deciding whether you want to call trySplit on the Spliterator again.
