However, since the zero of the computer’s time line has been defined to correspond to 00.00AM 1st Jan, 1970 UTC, converting an Instant to the time in the UTC time line doesn’t require any other information. Thus, you can say that an Instant object precisely corresponds to a date and time in UTC.  You can create an Instant for any given amount of time elapsed since epoch. For example, Instant.ofEpochMilli(1735711200000L).

LocalDate/LocalTime/LocalDateTime: These classes represent just a date, a time, or a date and time without any indication of the place for which that date/time is applicable. Think of it this way - if someone tells you that the President will address the nation at 8PM on 1st Jan 2025, you can’t really be sure exactly when you need to be ready to watch it because you haven’t been told the place for which this date and time is applicable. Without this information you may be too early or too late to watch the speech. Thus, you can’t take the information available in a LocalDateTime object and identify a specific point on the machine time line.

You can, at best, guess that this time is with respect to the region you are in and use that information to decide when to be ready to watch the President’s address. In that sense, it represents the date/time of the system on which the program is running. You will see several ways to create objects of these types in the next section.

ZoneId: A ZoneId identifies an area or a region, known as “Time Zone”, for which precise rules for determining the date and time of a day with respect to the UTC exist. These rules enable you to take a point on the machine’s timeline, i.e., an Instant object, and determine the exact date and time in that region at that instant. Note that a ZoneId object itself doesn’t contain any rule. It is just an id. JDK maintains the rules in an internal database of sorts (in ZoneRules class, actually, but you don’t need to worry about it) within its standard library, which can be retrieved using this id.
